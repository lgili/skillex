# Report Structure

Use the lightest structure that still supports the decision the document must
enable.

## Common Shapes

### Design Note

Use when the report explains why a design was chosen.

Recommended order:

1. Objective
2. Constraints and assumptions
3. Architecture or concept
4. Key equations or budgets
5. Tradeoffs
6. Risks and next actions

### Test Report

Use when the main question is what was tested and what passed.

Recommended order:

1. Objective and scope
2. Setup and instrumentation
3. Procedure summary
4. Results table
5. Deviations
6. Conclusion and follow-up

### Validation Report

Use when you must show compliance against requirements.

Recommended order:

1. Scope
2. Requirement matrix
3. Evidence package
4. Exceptions
5. Compliance statement
6. Open items

### Failure Analysis

Use when the report explains a technical incident or defect.

Recommended order:

1. Symptom and impact
2. Timeline
3. Evidence
4. Root-cause hypotheses
5. Confirmed cause or current leading cause
6. Corrective actions and verification plan

### Research-Style Summary

Use when the report is analytical and result-driven.

Recommended order:

1. Problem statement
2. Method
3. Model or equations
4. Results
5. Interpretation
6. Limitations
7. Conclusion

## Structure Rules

- Put the main question in the first page or first screen.
- Put the main result before supporting detail when possible.
- Keep sections narrow; one section should usually answer one technical point.
- Move raw supporting material to appendices when it breaks reading flow.
- Prefer tables for parameter inventories, requirement matrices, and result summaries.

## Good Compression Pattern

Bad pattern:

- long background
- long explanation
- conclusion hidden at the end

Better pattern:

1. Objective
2. Key result
3. Evidence
4. Interpretation
5. Risks and next actions
